# NightOps Core UX Redesign Specification (現場特化型・売上最大化UI)

本ドキュメントは、NightOpsを「現場で毎日使われるプロダクト（＝これがないと店が回らないインフラ）」へと昇華させるための、コア3画面（売上入力・日次締め・初回オンボーディング）のゼロベース再設計仕様書です。

## 共通UI・制約事項（Global Rules）
- **Theme**: 完全ダークモード (`bg-[#0A0E17]`, `text-[#F8FAFC]`)。現場の暗闇で眩しくなく、文字が浮き出るハイコントラスト設計。
- **Touch Target**: すべてのタップ可能要素（ボタン・リスト・入力欄）の高さを**最低 56px** に強制。
- **Thumb Zone**: アクションボタン、入力キーパッドはすべて画面下部（親指の可動域）に固定配置。画面上部は「表示のみ（Viewer）」に徹する。
- **Haptic Feedback**: 成功時・エラー時にはバイブレーション（Haptic API）または視覚的なシェイク/フラッシュを必ず実装する。

## 1. 売上入力画面 (Sales Input Screen)
【目的】最速で、正確に、ストレスなく売上金額を登録する。

### ① ワイヤーフレーム構造 (Text)
```text
[Header - 透過/固定]
← (戻る)        本日: 23:45 稼働中

[Display Area - 画面上部 30%]
・選択中キャスト: [ アイコン ] さくら ▼ (タップで変更)
・入力中の金額:
  ¥ 50,000 (巨大フォント・桁区切り自動)

[Details Area - スクロール可 20%]
＋ 内訳を追加 (指名/同伴/ドリンク等) ※オプショナル

[Undo Snackbar - 浮遊] ※直前に登録した直後のみ5秒表示
「さくら: ¥15,000 を登録しました」 [元に戻す]

[Keypad Area - 画面下部 50% 固定]
[ 1 ]  [ 2 ]  [ 3 ]
[ 4 ]  [ 5 ]  [ 6 ]
[ 7 ]  [ 8 ]  [ 9 ]
[ 00 ] [ 0 ]  [ ⌫ ]
[ ----- 登録 (スワイプ or 長押し) ----- ]
```

### ② コンポーネント一覧
- `DisplayValue`: 現在入力中の金額をリアルタイム表示する巨大なテキストコンポーネント。
- `CustomNumpad`: ネイティブキーボードを無効化し、画面下部に固定される巨大なグリッドボタン群。
- `SwipeToSubmit` / `HoldToSubmit`: 誤タップを防ぐための「スワイプ」または「長押し」で確定する登録ボタン。
- `UndoToast`: 登録直後に表示し、誤入力のリカバリを担保するスナックバー。

### ③ 状態設計
- **Idle**: 0円表示。キーパッド有効。
- **Typing**: 金額が入り、カンマが自動挿入。登録ボタンが青色（Active）に変化。
- **Loading**: 登録ボタンが「登録中...」に変わりスピナー表示（全キーパッド無効化）。
- **Success**: 画面全体がパッと緑にフラッシュ、金額が0にリセットされ UndoToast が出現。
- **Error**: 金額部分が赤く揺れる（Shake）。「通信エラー」等をトースト表示。

### ④ ユーザーフロー
1. 画面を開くと直前に操作したキャストがデフォルト選択されている（変更可能）。
2. カスタムテンキーで金額をタップ（例: `5` `0` `00`で50,000円）。
3. 画面最下部の巨大な「登録」ボタンを右へスワイプして 확 定（ミスタップ完全防止）。
4. 連続して、次の伝票を入力可能。

### ⑤ アニメーション仕様
- **入力時**: 数字が増える際、わずかなスケールアップ（Pop）で指の押下感を強調。
- **エラー時**: 金額表示部が左右に小刻みに震える（CSS `keyframes shake`）。
- **成功時**: `Hero` または `Check` アイコンがフワッと浮かび上がり消える。

### ⑥ なぜこの設計にしたか（ビジネス視点）
現場のボーイは暗い通路を歩きながら片手で伝票を処理します。標準のキーボードでは「0」の数を押し間違え、後から「売上が50万円も合わない！」というパニックを引き起こします。POSレジ専用機のようなカスタムテンキーと「00」ボタン、そしてスワイプ確定による「ミスタップの物理的排除」により、システムの信頼性を担保し、解約を防ぎます。

## 2. 日次締め画面 (Daily Closing Screen)
【目的】「絶対にズレない安心」を提供し、アプリへの依存度を最大化する。

### ① ワイヤーフレーム構造
```text
[Header]
× (閉じる)      日次締め処理

[Status Area - 画面上部]
本日のシステム売上総額
¥ 1,250,000

[Input Area - 中央]
レジの現金有高を入力してください
[ ¥ __________  ] (カスタムテンキーで入力)

[Validation Area - 中央下部]
差異: ¥ -50,000 (赤色・警告アイコン)
※ 0円になるまで締められません

[Keypad Area - 画面下部固定]
[ 1 ] [ 2 ] [ 3 ] ...
[ 締め処理を完了する (Disabled: グレー) ]
```
※入力して「差異：0円（緑）」になった瞬間、ボタンが「Active（青）」になる。

### ② コンポーネント一覧
- `TargetAmount`: システムが計算した本来あるべき金額（リードオンリー）。
- `ActualInput`: ユーザーが入力する実際のレジ現金有高。
- `DifferenceIndicator`: 差額を計算し、色（赤・緑）とアイコンで視覚的に警告・承認するUI。
- `LockableSubmitButton`: 差額が0円でないと絶対に押せないフェイルセーフボタン。

### ③ 状態設計
- **Idle**: 差異が「計算不能（未入力）」状態。ボタン非活性。
- **Typing (Mismatch)**: 差異が計算されるが、0円ではない状態。表示は赤文字。ボタン非活性。
- **Typing (Match)**: 差異が `0`円になった状態。表示が緑に発光し、「一致しました」マークが出る。ボタン活性。
- **Success**: 画面全体に「本日の業務が完了しました」という達成感のあるフルスクリーンアニメーション。

### ④ ユーザーフロー
1. 閉店後、金庫内の現金を数える。
2. アプリの締め画面を開き、数えた現金の額を入力。
3. 差額がマイナスなら「伝票の入力漏れ」、プラスなら「バックの払い忘れ等」に気づく。
4. 原因を突き止め、入力し直して「差異0円」にする。
5. 活性化した「締め処理」ボタンを押して帰宅。

### ⑤ アニメーション仕様
- **0円一致時**: 差額テキストが赤から緑へクロスフェードし、✅アイコンがポップアップ。画面の枠がうっすら緑に発光する。
- **締め完了時**: 紙吹雪（Confetti）エフェクトや、チェックマークが全画面で描画される「達成感」の演出。

### ⑥ なぜこの設計にしたか（ビジネス視点）
現金商売において、店長の最大のストレスは「現金と伝票が合わないまま帰ること」です。このUIはゲーミフィケーションの要素（差異0円＝クリア）を含み、「NightOpsを使えば現金が絶対にズレずに安心して帰れる」という圧倒的な体験を提供します。これこそが他社ツールへの乗り換え（コスト）を不可能にする最大のLTV向上要因となります。

## 3. 初回オンボーディング (Initial Onboarding Screen)
【目的】初日で「難しそう」から「これなら私でも使える」へマインドチェンジさせる。

### ① ワイヤーフレーム構造
```text
[Overlay Context - ダッシュボードは背景でBlur（ぼかし）状態]

[Progress Bar]
  === 1/3 ===

[Card UI - 中央]
「NightOpsへようこそ！
  まずは、今日出勤しているキャストを1名だけ登録してみましょう。」

[ Form Input: 名前 (例: さくら) ]
[ 登録して次へ (Blue Button) ]
```
※完了すると即座に STEP 2 (上で設計した売上入力画面のチュートリアル版) へ遷移。最後は STEP 3 (差異0円を体験する締め画面)。

### ② コンポーネント一覧
- `FocusOverlay`: 背面の複雑なダッシュボードを完全に隠し、今やるべき1つのタスクだけに集中させるスクリーン。
- `StepWizard`: 進行状況（1/3など）を示すプログレスバー。
- `InteractiveMock`: 本番と同じ「売上入力」「締め」のUIだが、特定の操作しか受け付けないサンドボックスコンポーネント。

### ③ 状態設計
- **Step 1 (Cast Add)**: 1人追加するまで次へ進めない。
- **Step 2 (Mock Input)**: チュートリアルの指示通りに「5,000円」と入力するまで進めない。
- **Step 3 (Mock Close)**: 現金有高「5,000円」と入力して差額を0にして締めるまで進めない。

### ④ ユーザーフロー
1. トライアル開始後、初回ログイン。
2. ダッシュボードのあらゆるボタンがロックされており、中央のカードしか操作できない。
3. 言われるがまま「A子」を登録。
4. 言われるがままテンキーで「10000」と叩いてスワイプ。
5. 言われるがまま現金「10000」を入れて差異0円にして完了。
6. 「おめでとうございます！今日からあなたの店舗の締め作業は、たったの3分になります。」が表示され、ダッシュボード解放。

### ⑤ アニメーション仕様
- **ステップ移行**: カードが左へスライドアウトし、次のステップカードが右からスライドイン（スムーズなページネーション）。
- **ユーザー誘導**: 入力すべきボタンの周りに、波紋のようなパルスアニメーション（Pulse Halo）を配置し、次にタップすべき場所を迷わせない。

### ⑥ なぜこの設計にしたか（ビジネス視点）
SaaSの死の谷は「初回ログイン直後の5分間」にあります。多機能な管理画面をいきなり見せると、水商売のITリテラシーでは90%が「後でゆっくり触ろう」と離脱し、二度と戻ってきません。
ダッシュボードを強制的に隠し、「キャスト追加 → 売上入力 → 差異0確定」という **NightOpsのコアバリューを強制的に3分以内で体験させる** ことで、「あ、これなら自分でもできる」というアハ体験（Aha moment）を確実に創出します。これにより、無料アクティベーションから有料課金へのCVRが飛躍的に向上します。
