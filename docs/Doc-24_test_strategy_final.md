## 1. テストフェーズ定義

| フェーズ | 対象・スコープ | フレームワーク | 実行タイミング |
|---|---|---|---|
| **単体 (Unit)** | 計算ロジック、ユーティリティ関数 | Jest | 各コミット (CI) |
| **統合 (Integration)** | APIエンドポイント、DBクエリ、Guard | Supertest + Jest | PR作成時 (CI) |
| **E2E** | 主要ユーザーシナリオ（出勤〜接客〜締め） | Playwright | mainデプロイ前 |
| **受入 (UAT)** | ステージング環境での手動テスト確認 | Manually | リリース前 |

---

## 2. 必須検証ケース1: テナント混線防止（Tenancy）

SaaS販売の絶対条件として以下の網羅テストをCIに組み込む。

- **T-01**: A店舗の Staff が B店舗の `storeCode` でログイン試行 → **403 Forbidden**
- **T-02**: A店舗の Admin が API 経由で B店舗の `salesSlipId` を指定し GET → **404 または 403** (TenantGuard)
- **T-03**: A店舗の Cast が JWT改ざん（テナントID書き換え）でリクエスト → **401 Unauthorized** (署名検証エラー)

---

## 3. 必須検証ケース2: 締めロック（CloseGuard）

データの不変性を守るためのロックテスト。

- **C-01**: `DailyClose` が `closed` な日の売上伝票に PUT リクエスト → **409 Conflict (CLOSE_001)**
- **C-02**: 同上日の勤怠打刻に対する手動修正 POST → **409 Conflict (CLOSE_001)**
- **C-03**: 締め未完了だが「未あがりキャスト」が存在する状態で締め実行 → **409 Conflict (CLOSE_003)**

---

## 4. 計算一致テスト（手計算との照合）

給与・売上の計算テスト（`packages/shared` に計10本以上実装済）から抜粋・拡張。

| ID | 計算種別 | 入力 | 期待値（式と手計算の合致） |
|---|---|---|---|
| M-01 | 小計算出 | [7000×3, 3000×2, 10000×1] | `Subtotal = 37,000` |
| M-02 | サービス税 | 小計 37,000, multiplier 1.32 | `RawTotal = 48,840` |
| M-03 | 丸め (繰上) | Raw 48,840 | `Total = 49,000` (+500繰上げ) |
| M-04 | 丸め (繰下) | Raw 48,499 | `Total = 48,000` |
| M-05 | 時給計算 | Hourly 3,000, 実働 5h30m | `Base = 16,500` (分単位合算) |
| M-06 | バック算出 | 本指名 1本(10%), 場内 2本(1000) | `Back = (1本売上×0.1) + 2,000` |
| M-07 | 補正伝票計算 | 小計 +1000 のCorrection生成 | `Raw = +1,320, Total = 合算後再計算` |
| M-08 | 歩合計算 | 個人の売上小計 100,000, 20% | `Base = 20,000` |
| M-09 | 日跨ぎ計算 | 23:00出勤, 02:00あがり | `Duration = 3h` (翌日計算にならない) |
| M-10 | トータル給与 | Base 16500 + Back 3000 | `TotalSalary = 19,500` |

## 5. リグレッション方針（アップデート想定）

- 計算ロジック（`packages/shared/src/calculations`）に変更を加えた場合、**過去の全てのテストケースを修正せず、追加テストとしてケースを作成**する。過去ルールに依存したデータが壊れないことを担保する。
- DBマイグレーション前には必ずステージングDBクローンでのドライランを実行する。
