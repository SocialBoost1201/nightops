# Doc-40: Sales Input Phase 1 Implementation Notes
## 1. 今回のスコープ (In Scope)
- `apps/mobile` ディレクトリの初期化と、ダークモード（#0A0E17）に特化したモバイル用UIテーマの適用。
- **CustomNumpad**: 標準キーボードを無効化し、片手での入力に特化した「巨大カスタムテンキー（00ボタン付き）」の実装。
- **SwipeToSubmit**: 物理的な「右スワイプ操作」でのみ登録を確定させるフェイルセーフ機能と、成功時/失敗時のハプティックバイブレーション（振動）および視覚的フィードバックの実装。
- **Autosave**: ネットワーク切断やアプリ再起動に備えた `localStorage` への数値自動保存（下書き保持）。
- **UndoToast**: 送信直後の「5秒間」だけ表示され、サーバーへ送信される前に直前の操作を取り消すことができるUndo機能の実装（Optimistic UIを活用）。

## 2. やらなかったこと (Out of Scope)
- 日次締め（Daily Closing）画面および初回オンボーディングの実装。
- 指名・同伴・ボトルバックなどの複雑な「売上内訳」入力UIの実装。
- 実際のバックエンドAPI（Supabase/Prisma）へのデータ永続化（今回は `console.log` でのモック通信）。
- 顧客管理・分析機能・複数キャストの動的切り替え。
- PCおよびタブレットサイズへのレスポンシブ最適化（今回は純粋なスマートフォンに特化）。

## 3. UX上の判断理由 (UX Design Decisions)
- **標準キーボードの排除と「00」追加**: 水商売の現場では「万」単位の入力が多く、0の押し間違いが致命的な売上ズレを引き起こします。POSレジライクな巨大キーパッドと「00」キーを設けることで、入力を数倍高速化＆正確化しています。
- **Swipe To Submit (フリック確定)**: 疲労や酔いで集中力が落ちている状態での「不意のタップ（二重登録）」を物理的に防ぐため、明確な意志が必要な「右スワイプ」をアクションのトリガーとして採用しました。
- **Undoの仕組みと5秒ディレイ**: API送信後にDBから物理削除する「ハードルーズなUndo」ではなく、送信自体を5秒遅延させる「ソフトなUndo」アーキテクチャを採用しました。これにより、SaaSとしてのデータ整合性（監査ログなど）を壊さずに安全な取り消し体験を提供します。

## 3.5. Phase 1 品質監査での追加修正事項 (Phase 1 Safety & Quality Fixes)
1. **「00」入力上限突破バグの修正**: 9,999,999円の上限判定を文字長ベースから数値比較ベースに変更し、異常値の入力によるインシデントリスクを遮断しました。
2. **Autosave（下書き）のキャスト混入防止**: 保存データをJSON化し `castId` と `updatedAt` を追加。復元時に同一キャストID判定と3時間の有効期限チェックを設けました。
3. **Undo中の二重操作防止**: Undo待機中（5秒間）に連続入力して状態不整合を起こさないよう、Undo表示中は入力基盤全体をUIレベルでロックしました。
4. **片手操作UXの向上 (長押し全消去)**: CustomNumpadの削除キーに、片手実働環境に特化した500msの長押し機能（全クリア）を付加しました。
5. **パフォーマンスとUIの視認性改善**: UndoToastのタイマー表示を `requestAnimationFrame` から Framer Motion の CSS Transform アニメーションへ移行し、再レンダー負荷をカット。SwipeToSubmitのつまみを白とネイビーカラー反転に再設計し、暗所でのコントラストを際立たせました。
6. **コード保守性の向上**: UIコンポーネント内に混在していた業務ロジック群を `useSalesInput` と `useSoftSubmit` にCustom Hooksとして抽出し、今後のテストおよびPhase 2 API接続の土台を整備しました。

## 4. 今後Phase2へ引き継ぐ項目 (Next Steps for Phase 2)
1. **日次締め画面・オンボーディングの実装**: 今回構築した `CustomNumpad` などの部品を流用し、コア業務機能のフロントエンドをフルコンプリートする。
2. **API接続（Backend Integration）**: 今回モックアップで作成した `handleSubmit` 内の `console.log` を、実際の `POST /api/sales` 等の通信ロジックへ載せ替える。
3. **売上内訳（Details）の拡張**: 総額入力（今回）の次に、バック額などの詳細な控除・加算情報を入力できるドロワーまたはモーダルを追加する。
