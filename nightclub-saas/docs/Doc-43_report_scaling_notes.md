# Doc-43 Report Scaling Notes

## 1. 現状実装のボトルネック
- `reports` は都度クエリ + メモリ集計方式（tenant/date条件付き）。
- 店舗数/伝票件数増加時に、以下がボトルネック化しやすい。
  - 日次/期間集計での全件走査
  - キャスト別ランキングのアプリ側集約
  - 同一期間への繰り返しアクセス

## 2. 将来候補と評価

### A. Daily Aggregates テーブル
- 内容: 日次×キャスト×指標（売上・本数・件数）を事前集計
- 長所: APIレイテンシ削減、計算の安定化
- 短所: 更新時の再集計設計が必要

### B. Monthly Aggregates テーブル
- 内容: 月次締め後の固定値を保存
- 長所: 月次ダッシュボード高速化
- 短所: 日次リアルタイム性には弱い

### C. Cache (Redis)
- 内容: 期間キーで report 結果を短TTLキャッシュ
- 長所: 実装が比較的軽い
- 短所: 無効化戦略を誤ると不整合

### D. Materialized View
- 内容: DB側で集計Viewを保持
- 長所: SQLで一貫した集計
- 短所: 運用/更新タイミング設計が必要

## 3. MVPで採用しないもの（今回）
- 大規模集計基盤（専用集計DB、非同期パイプライン）は採用しない。
- 理由: 現段階は API 契約統一と businessDate ルール固定が優先。

## 4. MVP後の推奨順序
1. Daily Aggregates（優先）
2. Redis Cache（読取集中時）
3. Monthly Aggregates（締め機能と連動）
4. 必要なら Materialized View へ拡張

## 5. 非機能要件メモ
- すべて tenant 単位でキー分離する。
- 集計結果キャッシュにも `tenantId` + `from` + `to` + `timezone` をキーへ含める。
- 監査上、締め済み期間の再計算方針を別途明記する。
