## 1. 目的
Doc-01〜Doc-19までの全設計書を統合検証し、アーキテクチャ・業務仕様・セキュリティの整合性を評価する。

---

## 2. 重大な矛盾・抜け・リスク一覧

### 🔴 High (高リスク・対応必須)
1. **月次確定時の給与計算パフォーマンス (Doc-15, Doc-12)**
   - **内容**: 月次給与確定時、全キャストの1ヶ月分の打刻・売上・バックを同期計算すると、APIレスポンスのSLO（500ms）を大きく超過するリスクがある。
   - **採択方針**: MVPでは1店舗20名規模を想定し「同期処理」として妥協するが、タイムアウト値を30秒に拡張し、UIはローディングスピナーで完了を待機する。非同期ジョブ化はPhase2へ回す。

### 🟡 Med (中リスク・運用カバー)
1. **初期パスワード忘れ時のリセットフロー (Doc-16)**
   - **内容**: 管理者が一括登録で初期パスワードを発行・強制変更させるが、キャストが変更後にパスワードを忘れた場合のフロー定義が薄い。
   - **採択方針**: MVPでは「管理画面からのパスワード強制リセット機能（JWT無効化連動）」のみ実装し、セルフリセット（メール認証）はPhase2へ回す。
2. **監査ログのデータ肥大化 (Doc-08, Doc-09)**
   - **内容**: 全ての更新・承認操作とログインで監査ログを出力するため、早期にテーブルが肥大化しクエリ遅延を招く。
   - **採択方針**: 監査ログテーブルは `tenant_id` と `created_at` の複合インデックスでスキャンを防ぐ。過去参照は90日制限とし、運用フェーズでパーティショニングを導入する（MVPでは何もしない）。

### 🟢 Low (低リスク・制限事項として許容)
1. **CSVインポート時のメモリ制約 (Doc-16)**
   - **内容**: 給与条件（CompensationPlan）やユーザー一括登録において、超巨大なCSVによるメモリ枯渇の懸念。
   - **採択方針**: アップロードAPIでファイルサイズ上限を 2MB（約1万行）にハードコードで制限する。

## 3. 整合性チェック結果

| 項目 | 判定 | 理由・評価 |
|---|---|---|
| **締めロック** | ✅ 整合 | `DailyClose.status=closed` と `CloseGuard` の連動でDBの更新をアプリレイヤで完全にブロック可能。Doc-15の設計はセキュアである。 |
| **補正方式** | ✅ 整合 | 締め後は `CorrectionSlip` / `CorrectionPayroll` による差分追加方式を徹底することで、改ざん防止とデータ不変性を両立できている。 |
| **監査ログ** | ✅ 整合 | `AuditLog` に `actorRole`, `correlationId`, `ipAddress` を持たせることで十分な追跡が可能。 |
| **RBAC / ABAC** | ✅ 整合 | 権限マトリクスに加え、`tenantId` と自己データ参照制御が ABAC（TenantGuard）として一貫して定義されている。 |
| **テナント分離** | ✅ 整合 | Prismaスキーマの全ビジネスデータに `tenant_id` を配置し、APIガードとインデックス設計を含め堅牢である。 |

## 4. MVPで危険な簡略化の指摘と代替案

#### 指摘1: 「トランザクション排他制御（二重更新）の省略」
UIからの連打等で同日に同一キャストが2回出勤打刻、または管理者が同一伝票を2回同時に保存するリスクがある。
**代替案**: 
UI側のボタン無効化に加え、APIレイヤーで必ず「打刻状態のSELECT」と「INSERT」を `SERIALIZABLE` または `UNIQUE` 制約に頼る。例えば出勤打刻は `(tenantId, accountId, businessDate)` のユニークチェックで防ぐ。

#### 指摘2: 「顧客統合機能（CustomerMerge）の非可逆性」
統合対象を間違えた場合、MVPでは元に戻す処理（Undo）が非同期・手動リカバリとなる。
**代替案**:
統合元も物理削除はせず論理削除 (`deletedAt`) としているため、データ消失はない。「統合取消」用APIを通し、ChangeRequestを介して管理者が論理削除フラグを落とす方式をMVPに内包させる。
